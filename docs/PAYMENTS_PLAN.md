# Payments Integration Plan

## Recommendation

Use Square as the first payment provider for the T&A Contracting portal.

Square is a good fit for the current portal because the business needs invoice/work-order payments, optional ACH for larger jobs, card payments, manual/offline payment confirmation, and a straightforward hosted checkout path before investing in a fully embedded payment UI.

## Why Square First

- Square already supports online payments, hosted checkout links, Web Payments SDK, Payments API, ACH, Cash App Pay, Apple Pay, Google Pay, invoices, and manual/offline payment records.
- Square-hosted checkout links reduce PCI and frontend complexity for the first production release.
- Square Web Payments SDK can be added later when the portal needs an embedded payment form.
- Square Payments API supports charging payment tokens generated by the Web Payments SDK and managing payment records server-side.
- Square pricing is competitive for online API card payments and capped ACH API payments, which matters for higher-value contracting invoices.
- Square Dashboard can still be used operationally while the portal integration matures.

## Alternatives Considered

### Stripe

Stripe is excellent for custom web checkout, subscriptions, saved payment methods, and developer flexibility. It may be a better fit later if the portal needs complex recurring billing, advanced payment method routing, or deeper custom checkout control.

### QuickBooks Payments

QuickBooks Payments can be useful if accounting synchronization becomes the primary goal. It is less ideal as the first portal payment integration unless bookkeeping workflow outweighs portal UX and API flexibility.

### PayPal / Venmo

PayPal can be useful as an additional payment option, but it should not be the primary integration for work-order invoices because Square better aligns with card/ACH/in-person contracting workflows.

## Initial Integration Approach

### Phase 1: Square-hosted payment links

Start with Square Checkout API payment links. This is the lowest-risk production path.

Portal flow:

1. Admin verifies completed work and moves the job request to `waiting_payment`.
2. The portal creates or updates the local invoice record.
3. Admin clicks a new action such as **Create Square payment link** on an open invoice.
4. Netlify Function calls Square Checkout API to create a hosted payment link for the invoice amount.
5. The portal stores the Square payment link ID, checkout URL, and status metadata locally.
6. Client invoice card shows a **Pay invoice** button that opens the hosted Square checkout page.
7. Square webhook confirms payment.
8. Webhook handler marks the local invoice paid, inserts a payment record, completes the job request, and writes an audit event.

Why this first:

- No card data touches our site.
- Minimal frontend payment code.
- Easiest path to production and support.
- Still supports payment status automation through webhooks.

### Phase 2: Square webhooks

Add webhook verification and payment status sync.

Recommended webhook events:

- Payment created/updated events to detect successful card/ACH payment.
- Order/payment-link related events if needed to correlate hosted checkout completion.

Webhook handling requirements:

- Verify Square webhook signature before processing.
- Use idempotency to avoid duplicate payment records.
- Store Square payment ID on the local `payments` row.
- Store Square checkout/payment link IDs on the local `invoices` row or a dedicated payment provider metadata table.
- Write audit events for payment completion and webhook errors.

### Phase 3: Embedded payment UI with Web Payments SDK

Add Web Payments SDK after the hosted checkout flow is stable.

Use embedded payments when:

- Clients should pay without leaving the portal.
- The portal needs ACH directly in the client invoice card.
- The business wants Apple Pay / Google Pay / Cash App Pay surfaced inside the portal.

Embedded flow:

1. Client opens invoice in portal.
2. Browser loads Square Web Payments SDK.
3. SDK collects card/ACH wallet payment details and returns a payment token.
4. Portal sends token to a Netlify Function.
5. Function calls Square Payments API with the token and invoice amount.
6. Function stores payment details locally and completes the invoice/job workflow.

## Data Model Additions

Recommended migration after choosing Square credentials:

### `invoices`

Add nullable fields:

- `payment_provider text default 'manual'`
- `provider_invoice_id text`
- `provider_checkout_id text`
- `provider_checkout_url text`
- `provider_status text`
- `provider_metadata jsonb not null default '{}'::jsonb`

### `payments`

Add nullable fields:

- `payment_provider text default 'manual'`
- `provider_payment_id text`
- `provider_status text`
- `provider_receipt_url text`
- `provider_metadata jsonb not null default '{}'::jsonb`

### Indexes

- `idx_invoices_provider_checkout_id`
- `idx_payments_provider_payment_id`

## Environment Variables

Add these before implementing Square API calls:

- `SQUARE_ENVIRONMENT=sandbox` or `production`
- `SQUARE_ACCESS_TOKEN=...`
- `SQUARE_APPLICATION_ID=...`
- `SQUARE_LOCATION_ID=...`
- `SQUARE_WEBHOOK_SIGNATURE_KEY=...`
- `SQUARE_API_VERSION=2026-01-22` or the Square API version selected for this project

Never expose the access token or webhook signature key to frontend code.

## Netlify Functions to Add

### `square-create-payment-link.mjs`

Admin-only endpoint.

Responsibilities:

- Validate admin invoice permission.
- Load the local open invoice.
- Create Square checkout payment link.
- Save provider IDs/URL/status to the local invoice.
- Return checkout URL for admin/client UI.

### `square-webhook.mjs`

Public endpoint with signature verification.

Responsibilities:

- Verify Square webhook signature.
- Parse event.
- Map Square payment/order/payment-link back to local invoice.
- Idempotently insert local payment row.
- Mark invoice paid when Square reports successful payment.
- Complete linked job request.
- Write audit events.

### Later: `square-create-payment.mjs`

Client endpoint for embedded Web Payments SDK tokens.

Responsibilities:

- Validate client owns invoice or admin has invoice permission.
- Accept tokenized `sourceId` from Square SDK.
- Create Square payment with idempotency key.
- Persist payment and invoice status locally.

## UI Changes Needed

### Admin invoice desk

- Add **Create Square payment link** for open invoices.
- Show provider status and checkout URL once generated.
- Keep **Confirm payment** manual flow for cash/check/offline payments.
- Add a clear note when payment will be confirmed automatically by Square webhook.

### Client invoice cards

- Show **Pay invoice** when `provider_checkout_url` exists.
- Show **Payment link pending** if an invoice is open but no link exists yet.
- Show paid/confirmed details after webhook or manual confirmation.

### Activity feed

- Add activity labels for `square.payment_link.created`, `square.payment.completed`, and `square.webhook.error`.

## Recommended Next Build Step

Implement Phase 1 in this order:

1. Add Square env vars to `.env.example` and docs.
2. Add invoice/payment provider metadata migration.
3. Add admin-only payment link function.
4. Add admin invoice desk action and client pay button.
5. Add webhook handler with signature verification.
6. Add tests for payment link creation, webhook idempotency, and invoice/job completion.
7. Update `docs/summary.md` after each completed milestone.

## Open Questions

- Should clients be able to pay by ACH on day one, or should ACH wait for embedded Web Payments SDK?
- Should Square payment links expire automatically after a certain number of days?
- Should partial payments/deposits be supported, or only full invoice balances?
- Should invoices support multiple payments against one invoice?
- Should payment provider metadata live directly on `invoices`/`payments` or in a separate `payment_provider_events` table?

## Decision

Proceed with Square-hosted checkout links first, then Square webhooks, then embedded Web Payments SDK once the hosted flow is stable.
